## Summary

The server broadcasts `status_tick` WebSocket events when tick damage (Burn, Poison, Badly Poisoned, Cursed) is applied at turn end, but the client-side WebSocket handler (`composables/useWebSocket.ts`) has no case for `status_tick`. Events are silently dropped.

State sync still works because `gm/index.vue` broadcasts a full `encounter_update` after `nextTurn()` completes, which includes post-tick HP values. So this is not a data correctness issue.

## Impact

Without a `status_tick` handler, Group/Player views cannot show toast notifications or visual feedback specifically for tick damage events (e.g., "Pikachu lost 7 HP from Burn"). This will matter more as P1/P2 of feature-010 add save checks and additional status automation.

## Affected Files

- `app/composables/useWebSocket.ts` — add `case 'status_tick':` handler
- `app/server/routes/ws.ts` — event already registered and broadcasting (line 349-354)

## Suggested Fix

Add a `case 'status_tick':` handler in `useWebSocket.ts` that triggers a toast/notification showing which combatant took how much tick damage from which condition. The event payload from the server includes `tickResults` with per-entry damage, condition, and combatant info.
