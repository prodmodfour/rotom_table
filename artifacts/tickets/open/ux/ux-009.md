# ux-009: Show type-immunity IMMUNE tags on all statuses proactively

## Problem

In StatusConditionsModal, the IMMUNE tag for type-based status immunities only appears when a status is both checked in the UI AND is a new addition. The GM has no visual cue about which statuses the target Pokemon is immune to when browsing the available options.

## Current Behavior

- GM opens StatusConditionsModal for a Fire-type Pokemon
- All status checkboxes appear identical (no visual distinction)
- GM checks "Burned" → IMMUNE tag appears + warning banner
- Server rejects with 409 if submitted without override

## Expected Behavior

- GM opens StatusConditionsModal for a Fire-type Pokemon
- "Burned" checkbox immediately shows an IMMUNE indicator (e.g., dimmed text + small tag)
- GM can still check it and use "Force Apply (GM Override)" if needed
- Proactive visibility reduces trial-and-error for GMs unfamiliar with all type immunities

## Affected Files

- `app/components/encounter/StatusConditionsModal.vue` (lines 117-122, `getImmuneLabel()` function)

## Suggested Fix

Modify `getImmuneLabel()` (or add a new computed) to always show IMMUNE tags for statuses where the entity's types grant immunity, regardless of checkbox state. The existing `isNewAndImmune` computed only checks new additions — add a parallel `isImmune` check that applies to all statuses.

## Impact

- UX improvement only — no game logic or correctness impact
- Server-side 409 enforcement still prevents incorrect state regardless
- Helps GMs who are less familiar with PTU type-immunity rules
