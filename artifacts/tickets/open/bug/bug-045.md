# bug-045: CapturePanel receives entityId instead of combatantId for trainer action economy

## Summary

CombatantCard.vue's `availableTrainers` computed uses `c.entityId!` (the HumanCharacter entity ID) for the trainer options, but CapturePanel passes this as `trainerCombatantId` to the action consumption endpoint, which expects a combatant ID. This means the standard action consumption after a capture attempt silently fails because the endpoint cannot find the combatant by entity ID.

## Problem

The capture roll itself is unaffected — the actual capture succeeds or fails correctly. However, the post-capture action economy tracking (consuming the trainer's Standard Action for the turn) fails silently. The `useCapture()` composable sets a `warning` ref when this happens, but code-review-295 H2 notes that CapturePanel does not display this warning.

The practical effect: after a capture attempt, the trainer's Standard Action is NOT consumed, allowing them to take additional actions in the same turn.

## Affected Files

- `app/components/encounter/CombatantCard.vue` — `availableTrainers` computed builds options with `entityId` instead of combatant `id`

## Suggested Fix

In `availableTrainers`, use the combatant's `id` (combatant ID) instead of `entityId` (entity ID):

```typescript
// Current (wrong):
{ label: displayName, value: c.entityId! }
// Fixed:
{ label: displayName, value: c.id }
```

## Notes

- Pre-existing issue from P1/P2 integration, not a P2 regression
- The fix for code-review-295 H2 (display warning) should also be applied, but this root cause fix prevents the warning from occurring
- Source: rules-review-268 Observation 2, cross-referenced with code-review-295 H2
