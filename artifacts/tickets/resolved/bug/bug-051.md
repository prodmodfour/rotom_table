# Tickets index generator uses frontmatter status instead of directory path

## Summary

The `generateTicketsIndex()` function in `regenerate-artifact-indexes.mjs` was filtering tickets by their YAML frontmatter `status` field rather than using the directory location (`tickets/open/`, `tickets/in-progress/`, `tickets/resolved/`) as the source of truth. This caused severe count mismatches that propagated into the orchestrator survey and all downstream planning.

## Problem

When tickets are moved between status directories (e.g., from `in-progress/` back to `open/` during collection), their YAML frontmatter `status` field is not always updated to match. The index generator read the stale frontmatter and classified tickets incorrectly:

- **tickets/_index.md** reported 27 open (actual: 48) and 122 in-progress (actual: 8)
- 21 open tickets were invisible to the orchestrator survey
- 114 resolved tickets were incorrectly counted as in-progress

Both `generateTicketsIndex()` and `generateGlobalIndex()` had this bug.

## Root Cause

Line 288: `const open = allTickets.filter((t) => t.status === 'open')` — reads `t.status` from frontmatter instead of from the directory path that was already available in the loop variable.

## Resolution

Fixed in commit f47f9e30: generator now overrides `status` with the directory name when building the ticket list (`allTickets.push({ ...t, category: cat, status })`). Both `generateTicketsIndex()` and `generateGlobalIndex()` were patched.

Additionally, 7 open tickets with stale frontmatter were corrected (commit 34e4b37a), and 5 fully-implemented tickets stuck in `in-progress/` were moved to `resolved/` (commit 3d5c8b37).

## Remaining Risk

The fix addresses the generator, but the root cause — ticket moves not updating frontmatter — can still cause confusion for tools that read individual ticket files. Consider adding a pre-commit hook or collection-step validation that ensures frontmatter status matches directory location.
