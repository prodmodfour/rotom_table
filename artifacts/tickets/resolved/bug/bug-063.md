## Summary

`Math.max(modifiedSpeed, 2)` is applied after the Slowed condition halves speed. For a Pokemon with base 4m speed: Slowed halves to 2m, then floor keeps it at 2m — Slowed has zero effect. Worse: if speed would drop to 1m, the floor raises it back to 2m, effectively boosting speed.

## Suggested Fix

Apply the Speed CS floor of 2 BEFORE status condition modifiers (Slowed/Stuck), or use a different floor logic that respects condition penalties.

## Impact

Slowed condition is ineffective or counterproductive on slow Pokemon.

## Resolution Log

### Fix (2026-03-06)

**Root cause:** Movement modifier application order was wrong. Speed CS (with floor of 2) was applied after Slowed halved the speed, preventing Slowed from reducing speed below the CS floor.

**PTU rule verification:**
- PTU 1.05 p.700: Speed CS negative "may never reduce [movement] below 2" — floor is specific to CS penalties
- PTU 1.05 p.1718: Slowed "Movement halved (minimum 1)" — independent condition, not subject to CS floor

**Fix:** Reordered application: Speed CS (with floor of 2) is now applied BEFORE Slowed/Thermosensitive conditions. This allows Slowed to meaningfully halve the CS-adjusted speed below 2.

**New order:** Stuck → Tripped → Speed CS (floor 2) → Slowed (min 1) → Thermosensitive → Sprint → final minimum (1 if base > 0)

**Duplicate code path:** `intercept.service.ts` had a full copy of the movement modifier logic (`getCombatantSpeed`). Replaced with a thin wrapper calling the canonical `applyMovementModifiers` from `~/utils/movementModifiers.ts`.
