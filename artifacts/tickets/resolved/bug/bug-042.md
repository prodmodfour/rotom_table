# bug-042: release-hold.post.ts creates duplicate turnOrder entry

## Summary

When a held combatant is released, `turnOrder.splice(currentTurnIndex, 0, combatantId)` inserts them at the current position, but their original entry in the turn order was never removed when they held. After release, the combatant's ID appears twice in the turn order.

For League Battles this is acceptable (turn order rebuilt at phase transitions). For Full Contact battles, the turn order persists across rounds. At round reset, `resetCombatantsForNewRound` resets `hasActed = false` for all combatants but does NOT deduplicate the turn order â€” the combatant would get two turns in every subsequent round.

## Affected Files

- `app/server/api/encounters/[id]/release-hold.post.ts` (line 85)

## Suggested Fix

**Note:** The original suggested fix (indexOf after insertion) was BLOCKED by code-review-268 CRIT-001.
The indexOf(combatantId, currentTurnIndex + 1) pattern only searches AFTER the insertion point.
For hold-release, the original entry is typically BEFORE currentTurnIndex, so it was never found.

Correct approach â€” remove original BEFORE inserting, adjust currentTurnIndex:

```typescript
const originalIndex = turnOrder.indexOf(combatantId)
if (originalIndex !== -1) {
  turnOrder.splice(originalIndex, 1)
  if (originalIndex < currentTurnIndex) {
    currentTurnIndex--
  }
}
turnOrder.splice(currentTurnIndex, 0, combatantId)
```

**Why this differs from priority.post.ts:** For priority, the combatant hasn't acted yet, so their
original position is always AFTER currentTurnIndex â€” the post-insert search works. For hold-release,
the combatant held at an earlier turn and is released later, so the original is BEFORE currentTurnIndex.

## Impact

- Full Contact battles: combatant gets two turns per round after hold-release
- League Battles: unaffected (phase transitions rebuild turnOrder)
- Must be fixed before feature-016 P2
