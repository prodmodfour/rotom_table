## Summary

`useCapture.ts` (line ~192) references `/api/encounters/${encounterId}/action` to consume the Standard Action when a capture is performed during an encounter. This endpoint does not exist in the API routes. The code is wrapped in try/catch so it doesn't crash, but the Standard Action is never consumed.

## Impact

When a capture attempt is made during an encounter, the combatant's Standard Action is not consumed. They can continue to take additional Standard Actions after throwing a Poke Ball. This is a PTU action economy violation.

The `warning` ref gets set ("Capture succeeded but standard action was not consumed") but this only appears as a client-side warning, not as enforcement.

## Affected Files

- `app/composables/useCapture.ts` — line ~192, references non-existent endpoint
- Server: no `/api/encounters/[id]/action.post.ts` endpoint exists

## Suggested Fix

Either create a dedicated `/api/encounters/[id]/action.post.ts` endpoint that consumes the current combatant's Standard Action, or integrate action consumption into the existing capture attempt flow (e.g., in `attempt.post.ts` when `encounterContext` is provided).

This should be resolved before the player capture UI goes live (feature-023 P1).

## Resolution Log

### Fix: Create `/api/encounters/[id]/action.post.ts` endpoint

**Commit:** `225c16a6` — `feat: add action consumption endpoint for encounters`

**Files changed:**
- `app/server/api/encounters/[id]/action.post.ts` (new) — 83 lines

**Approach:** Created the missing endpoint rather than inlining into `attempt.post.ts`. Three callers reference this endpoint:
1. `app/composables/useCapture.ts:209` — consume Standard Action after capture attempt
2. `app/stores/encounter.ts:1041` — generic `useAction()` store method
3. `app/composables/usePlayerRequestHandlers.ts:87` — consume Standard Action on missed ball throw

The endpoint is a general-purpose action consumer (not capture-specific), so a standalone endpoint is the correct fix. It follows the same pattern as `pass.post.ts`: loads encounter via `loadEncounter()`, finds combatant via `findCombatant()`, validates the action hasn't already been used, sets the appropriate `turnState` field to `true`, saves via `saveEncounterCombatants()`, and returns the encounter response via `buildEncounterResponse()`.

**Validation:** The endpoint rejects requests where the action has already been consumed (400 error), preventing double-use.
